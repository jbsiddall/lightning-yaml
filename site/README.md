# Lightning YAML — documentation site

The marketing + docs website for **Lightning YAML**, built with **Astro + Starlight**.
It is a **static** site (`dist/` of pre-rendered HTML/CSS/JS — no SSR, no runtime
server) with a flashy landing page, an API reference generated from in-source TSDoc,
hand-written guides, a benchmarks page rendered as build-time inline SVG, and a research
section.

This is a **self-contained nested project**: it has its own `package.json` and
`pnpm-lock.yaml` and does not touch the root parser project. Run every command with
`pnpm --dir site …` from the repo root (or `cd site` first).

## Quick start

```bash
pnpm --dir site install
pnpm --dir site dev       # local dev server (runs starlight-typedoc too)
pnpm --dir site build     # → site/dist/  (static; what CI/Vercel ships)
pnpm --dir site preview   # serve the built dist/ locally
```

## API reference (starlight-typedoc)

The API pages are generated **from TSDoc in the source** during `astro build`/`dev`
(not `preview`) by [`starlight-typedoc`](https://github.com/HiDeoo/starlight-typedoc),
which wraps TypeDoc + `typedoc-plugin-markdown`. Output goes to
`src/content/docs/api/` (gitignored — regenerated every build).

Today the entry point is a **documented facade**, `src/lib/lightning-yaml.ts`, which
declares the full intended public surface (`parse` / `parseAll` / `stringify` /
`isValid`, `ParseOptions` / `StringifyOptions` / `YAMLWarning`, `Schema`,
`YAMLParseError`) with thorough `@param` / `@returns` / `@throws` / `@example` /
`@defaultValue` / `@see`. This makes the whole pipeline demonstrable end-to-end.

**To document the real library instead**, point the entry point at the parser source
in `astro.config.mjs`:

```js
starlightTypeDoc({
  entryPoints: ['../src/index.ts'],   // was: ['./src/lib/lightning-yaml.ts']
  tsconfig: '../tsconfig.json',        // the parser's tsconfig
  // …
})
```

Do this once `../src/index.ts` exports the same public surface with equivalent TSDoc.
No other change is needed — the sidebar group, routing, and styling are unaffected.

## Benchmarks + the `benchmark-data` orphan branch

The benchmarks page reads YAML at **build time** (`?raw` import + the `yaml` package)
and renders **static inline SVG** — zero client JS by default.

Benchmark results are split by suite, **one file per suite**, each an **append-only
stream of YAML documents** (one `---` document per run). This lets CI record a run by
*appending* a document — never rewriting — so runs accumulate with no read-modify-write
and no merge conflicts:

| file               | suite       | measures                                            |
| ------------------ | ----------- | --------------------------------------------------- |
| `speed.yaml`       | speed       | per-iteration parse + stringify time (`ns/iter`)    |
| `memory.yaml`      | memory      | peak RSS (MB) + V8 heap delta (KB)                  |
| `conformance.yaml` | conformance | yaml-test-suite pass rate (%)                       |

A new benchmark suite (e.g. a future `bundle-size.yaml`) gets its **own** file. The
site reads every document and renders the **newest** by default.

**Two locations, one schema:**

- `site/src/data/benchmarks/*.yaml` — a **committed sample** so `dev`/`build` work
  locally with no data branch present.
- Orphan branch **`benchmark-data`** — the source of truth, holding **only** these YAML
  files. Keeping it orphan means high-frequency data commits never pollute `main`.

CI checks out both and overlays the branch data over the sample **before** building
(see `.github/workflows/deploy.yml`). Because the two share one schema, local and
production render identically.

### Re-creating / seeding the orphan branch

The branch already exists and is seeded. To recreate it from scratch (Git ≥ 2.42):

```bash
git worktree add --orphan -b benchmark-data ../benchmark-data-wt
cp path/to/{speed,memory,conformance}.yaml ../benchmark-data-wt/
cd ../benchmark-data-wt
git add -A && git commit -m "chore: seed benchmark-data"
git push -u origin benchmark-data
cd - && git worktree remove ../benchmark-data-wt
```

A CI benchmark job appends a run like:

```bash
{ echo '---'; ./emit-speed-run-as-yaml.sh; } >> speed.yaml
git commit -am "bench(speed): run for $GITHUB_SHA" && git push
```

## Deployment — Vercel via GitHub Actions

Production ships **only** through the visible workflow in
`.github/workflows/deploy.yml` (prod on `main`, preview on PRs) using the Vercel CLI —
**not** Vercel's git auto-deploy. Disable auto-deploy in the Vercel project so the
Actions log is the single auditable deploy path.

On pull requests the `deploy` job posts (and updates) a **sticky comment with a
clickable preview URL** so you can open the built site straight from the PR. This needs
`pull-requests: write` on the job (already set) — for same-repo PRs the built-in
`GITHUB_TOKEN` covers it; PRs from forks get a read-only token and skip the comment.

**Vercel project settings:** set **Root Directory = `site`** so `vercel build`/`deploy`
operate on this nested project.

**Required CI config** (GitHub → Settings → Secrets and variables → Actions). Only the
token is sensitive, so the two IDs are plain **variables** and the token is a **secret**
(the workflow reads them from the `vars` and `secrets` contexts respectively — they are
separate and do not fall through to each other):

| name                | kind         | what it is / where to find it                                  |
| ------------------- | ------------ | -------------------------------------------------------------- |
| `VERCEL_API_KEY`    | **secret**   | a Vercel **token** — Vercel → Account Settings → Tokens (used as `--token`) |
| `VERCEL_ORG_ID`     | **variable** | opaque `team_…` / user id — `.vercel/project.json` after `vercel link` (`orgId`); NOT the `jbsiddalls-projects` slug |
| `VERCEL_PROJECT_ID` | **variable** | opaque `prj_…` — `.vercel/project.json` after `vercel link` (`projectId`); NOT the `lightning-yaml` project name |

An optional **GitHub Pages** fallback lives in `.github/workflows/pages.yml`
(`workflow_dispatch`); it builds with a base path and uploads `dist/`. Vercel is primary.

## Conventions

- **SSG only** — Astro's default `output: 'static'`; do **not** add an SSR adapter.
- **Minimize client JS** — prefer static `.astro`; use islands (`client:*`) only where
  interactivity is genuinely required.
- **All prose is Markdown/MDX** under `src/content/docs/` so it's easy to edit.
