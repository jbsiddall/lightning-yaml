Speed, memory, and spec-conformance for **Lightning YAML** against the other parsers this
repo benchmarks itself against: `js-yaml`, `yaml` (eemeli), and the `JSON` baseline where a
workload is JSON-shaped enough for it to compete. Every chart below is a static image —
inline SVG rendered at build time, no client JavaScript.

<p style="color:var(--sl-color-gray-3);font-family:var(--sl-font-mono);font-size:0.82em">{provenanceLine}</p>

These numbers track real runs, not a hand-captured snapshot: CI publishes each run to the
orphan `benchmark-data` branch, and the deploy workflow overlays that history onto this
page at build time — the charts above render the newest published document per suite.

<ChartCard

The **js-yaml (tuned)** series is the same library dumping with the lean `CORE` schema
instead of its heavyweight default (YAML 1.1 plus timestamp/binary/omap/set). On
JSON-shaped data that runs ~1.3–1.5× faster and round-trips identically — so it, not the
default, is the fair yardstick for how fast js-yaml *can* stringify this data. It's shown
on the stringify chart only: the rich workloads genuinely need the full schema, where the
"tuned" config would be identical to the default.

## Conformance

<ChartCard
  heading="YAML test-suite conformance"
  note={`Higher is better. Pass rate across ${conformance.suite_total} cases from the official yaml-test-suite.`}
  svg={conformanceSvg}
/>

`lightning-yaml` passes 97.6% of the suite — including all 91 negative (should-fail)
cases — ahead of `yaml` at 97.1% and `js-yaml` at 94.9%. Merge keys (`<<`), absent from the
test corpus, are the only unimplemented construct.
